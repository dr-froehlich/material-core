### REQ-017: Wire brand symlinks into `<project>/slides/` so logo, favicon, and theme resolve at slide render time

- **Status:** OPEN
- **Added:** 2026-05-21
- **Completed:** â€“
- **Verified by:** â€“
- **Depends on:** REQ-013, REQ-014
- **Description:**

  Slides projects scaffolded with `matctl project add --slides` end up
  with a `slides/_quarto.yml` that references `brand-assets/THD-logo.png`
  (per-brand placeholder in `_brand_resolve.brand_placeholders`) and a
  theme chain of `../brand.scss`, `../shared/base.scss`. The theme paths
  resolve correctly (Quarto handles `../` for theme files), but the
  `logo:` path is rendered into the HTML output verbatim as
  `<img src="brand-assets/THD-logo.png">` and resolved by the browser
  relative to `slides/_output/01-workflow.html`. The browser asks for
  `_output/brand-assets/THD-logo.png` â€” which Quarto never copied,
  because at render time it cannot find a `slides/brand-assets/`
  directory either (`matctl link` only fans out at the project root, not
  into subdirectories).

  Result: every slides project across every brand renders without a
  logo. The bug has been silently shipping since `--slides` was
  introduced (REQ-013, v0.7.0); it surfaced only when the
  `heroes-it-concept` project â€” a single-doc-with-slides combination
  that previously had no slides â€” was rendered end-to-end on 2026-05-21
  and a 404 for `brand-assets/THD-logo.png` showed up in the preview
  server log.

  Same mechanism likely affects favicon (`favicon: brand-assets/...`),
  though favicons fail less visibly. Theme SCSS is unaffected only
  because the slides template already uses `../` for theme paths;
  changing the `logo:` path the same way is not a fix on its own,
  because at runtime the browser still cannot reach above
  `slides/_output/`. The file has to be discoverable from inside
  `slides/` so Quarto copies it to `_output/brand-assets/...`.

  Scope:

  - Symlink wiring inside `slides/` so all brand-referenced paths
    (`brand-assets/`, plus `_brand.yml`, `brand.scss`, and `shared/` if
    needed for completeness) resolve identically to the project root.
  - Apply at three points: `matctl project add --slides` (scaffold),
    `matctl link` (retrofit existing projects), `matctl project modify
    --brand` (re-seat on brand change).
  - Retrofit notes for the `material` content repo so existing slides
    projects pick up the wiring on the next `matctl link --force`.

  Non-goals:

  - Changing the logo path itself (e.g. to `../brand-assets/...`). The
    browser cannot reach outside `_output/`; only Quarto-side resource
    copying solves this, and that requires the file to be discoverable
    at the path Quarto sees.
  - Restructuring brand-assets storage. The pkg-side
    `material_core/brands/<brand>/assets/` layout stays put.

- **Acceptance criteria:**
  - [ ] `matctl project add <name> --slides ... --brand <brand>` produces a `<project>/slides/brand-assets` symlink resolving to the project's brand assets (i.e. through the project-root `brand-assets` symlink â†’ `material_core/brands/<brand>/assets`).
  - [ ] `matctl link` at the repo root and inside a project directory creates the same `slides/brand-assets` symlink whenever `slides/` exists.
  - [ ] `matctl unlink` removes the `slides/brand-assets` symlink alongside the project-root symlinks.
  - [ ] `matctl project modify --brand <newbrand>` reseats `slides/brand-assets` (via unlink + link).
  - [ ] After `matctl link --force` on any existing slides project (chapters or single structure), rendering the slides produces no 404s for `brand-assets/*` and the logo appears in the rendered slide HTML.
  - [ ] Whatever set of slides-side symlinks is wired is documented in `CLAUDE.md` matctl section and in `_brand_resolve.py` module docstring.

- **Notes:**

  Diagnostic context (2026-05-21):

  - Confirmed via headless Chromium inspection of `heroes-it-concept/slides/_output/01-workflow.html`: the rendered HTML contains `<img src="brand-assets/THD-logo.png" class="slide-logo">`, and the preview server logs a 404 for that path. Adding `slides/brand-assets â†’ ../brand-assets` manually on the user's project resolved the 404 and the logo appeared in the bottom-right of every slide.
  - The current `_brand_resolve.link_project` symlinks `_brand.yml`, `brand.scss`, `brand-assets`, `shared` at the project root only. The slides _quarto.yml uses `../brand.scss`, `../shared/base.scss` (which works because Quarto evaluates theme paths server-side) but `logo: brand-assets/...` and presumably `favicon: brand-assets/...` (which gets baked into the HTML and resolved client-side relative to the slide output, so the file must live under `slides/`).
  - Minimum viable fix: replicate just `brand-assets` into `slides/`. The other three slides-side symlinks (`_brand.yml`, `brand.scss`, `shared/`) are not needed for the runtime path resolution since the slides _quarto.yml uses `../` for them, but adding them for symmetry would make the slides subdirectory self-contained (useful if anyone ever runs `quarto preview slides/01-*.qmd` while CWD is `slides/`). Decide during the planning step.
  - Upstream impact: every existing slides project across every consumer of `material-core` is affected. Release notes for the resulting version bump should call out the required `matctl link --force` after upgrade.
