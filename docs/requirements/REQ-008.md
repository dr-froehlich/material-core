### REQ-008: Group lifecycle, titles, and `modify` commands in matctl

- **Status:** OPEN
- **Added:** 2026-04-14
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-007
- **Description:**

**Problem:** REQ-007 introduces groups as a string label attached to course
and doc entries. Three gaps remain:

1. **Typo risk.** A `--group mk4-26` flag on `course add` and `--group mk4-26`
   on `doc add` must reference the same string character-for-character or
   the projects silently diverge into two different URL paths. Nothing today
   catches a typo at authoring time.

2. **No human-readable titles in the manifest.** `matctl course add --title
   "..."` substitutes the title into the scaffolded Quarto files but does
   not record it in `projects.yml`. The auto-generated group landing page
   (REQ-009) needs per-project titles *and* a group title — neither of which
   the manifest currently stores.

3. **No way to change a title or group after creation.** The only way to
   correct a typo or retitle a project today is to remove and re-add it,
   which loses history and requires Git work. A `modify` subcommand is
   missing independently of group work.

**Solution:** Promote the group to a first-class manifest entity with a
lifecycle (`matctl group add/remove/modify`), write human-readable titles
into `projects.yml` on every `add`, and add a `modify` subcommand to each of
`course`, `doc`, and `group`.

---

### 1. `projects.yml` schema additions

Add a new entry type `group` and a `title` field on all entry types:

```yaml
projects:
  - name: mk4-26
    type: group
    title: "Mikrocomputertechnik 4 — WS 2026"

  - name: digital-und-mikrocomputertechnik
    type: course
    group: mk4-26
    title: "Digitaltechnik und Mikrocomputertechnik"

  - name: esp-survival-guide
    type: doc
    group: mk4-26
    title: "ESP Survival Guide"

  - name: unrelated-standalone-doc
    type: doc
    title: "Unrelated Standalone Doc"
```

Schema rules:

- `type: group` entries have `name` and `title`. No `group:` field (groups
  cannot nest).
- `type: course` and `type: doc` entries have `name`, `type`, `title`, and
  optionally `group`.
- `title` is required on all entries (written by `matctl add`; defaulted to
  title-cased slug if `--title` omitted, matching today's template behaviour).
- All `name` values share a single namespace: a group, a course, and a doc
  cannot have the same `name`. `matctl add` rejects collisions.

---

### 2. `matctl group` command group — new

```bash
matctl group add <name> --title "..."
matctl group remove <name> [--yes]
matctl group modify <name> --title "..."
```

Behaviour:

- **`group add`** writes a `type: group` entry to `projects.yml`. Purely a
  manifest operation; no on-disk footprint. Fails on name collision.
- **`group remove`** fails with a clear error if any `course` or `doc` entry
  has `group: <name>`. Prints the list of dependents. Otherwise deletes the
  manifest entry.
- **`group modify`** updates only `--title`. No other fields are modifiable.

---

### 3. `matctl course` / `matctl doc` — `--title` in manifest, new `modify`

Existing behaviour (REQ-007 + REQ-004 + REQ-005):

- `course add <name> [--title "..."] [--subtitle "..."] [--group <g>]`
- `doc add <name> [--title "..."] [--group <g>]`

Changes:

- On `add`, `title` (and `group` if provided) are written to `projects.yml`
  in addition to template substitution as today.
- **`course/doc add --group <g>`** requires `<g>` to exist in `projects.yml`
  as a `type: group` entry. Otherwise fails with a clear error: the group
  must be created with `matctl group add` first.
- **New subcommand `modify`:**
  ```bash
  matctl course modify <name> [--title "..."] [--group <g>]
  matctl doc    modify <name> [--title "..."] [--group <g>]
  ```
  At least one of `--title` / `--group` must be provided. `--group ""`
  (empty string) removes the grouping (project becomes root-level).

---

### 4. `modify --title` — write-through to scaffolded files

When `modify --title` is called on a course or doc, matctl rewrites the
`title:` key in:

- `<name>/_quarto.yml`
- `<name>/index.qmd` (front matter)

Implementation: YAML-aware round-trip via `ruamel.yaml` (same library already
used for `projects.yml`). Only the `title:` key is touched; all other keys,
comments, and formatting are preserved.

Matctl owns the `title:` key in these files. A subsequent `matctl course
modify --title` will overwrite hand-edits to those specific keys. 
`matctl modify --title`; they can edit `projects.yml` directly (carefully) or
hand-edit the scaffolded files in full knowledge.

`group modify --title` only touches the manifest; groups have no scaffolded
files of their own (the landing page from REQ-009 is regenerated from the
manifest every time, not hand-editable).

---

### 5. `modify --group` — meta-operation

Changing a project's group is a meta-operation composed of atomic steps:

1. Validate the new group exists (unless empty string).
2. Update the `group` key (or remove it) in the project's manifest entry.
3. Trigger index-page regeneration for both old and new group (per REQ-009).
4. Print a message warning the user that the deploy path on Netcup will
   change on next CI run, leaving stale content at the old path which must
   be cleaned up manually — same pattern as `course/doc remove`.

No automatic remote cleanup. No automatic token management (tokens are dumb
KV keys; re-issue or revoke as needed).

---

### 6. Rename operations — deferred

Renaming a group, course, or doc (changing its `name`) is **not** part of
this requirement. Rename cascades across directory names, deploy paths, and
issued tokens; supporting it cleanly requires its own design.

A known future use case: migrating a whole group from one slug to another
(e.g. `mk4-26` → `mk4-27`) at the start of a new term. Given the atomic
operations defined here, this can be achieved as a sequence:

1. `matctl group add mk4-27 --title "..."`
2. For each project in `mk4-26`: `matctl course|doc modify <name> --group mk4-27`
3. `matctl group remove mk4-26`

A future `matctl group rename <old> <new>` could wrap this sequence plus
token migration; open as a new requirement when the need arises.

- **Notes:**

- REQ-009 builds on the manifest shape defined here: every entry has a
  `title`, groups have a known title, and any structural change to a group's
  membership flows through one of the `matctl` commands specified here.
- The `subtitle` field on courses is not promoted to the manifest in this
  requirement — it stays template-only (SCSS substitution) as today.
  Revisit if the landing page (REQ-009) ever needs to display it.
- Validation is bidirectional: `course/doc add --group` requires the group
  to exist; `group remove` requires no projects reference it. Together these
  keep the manifest consistent at all times without a separate integrity
  check.
- Write-through of `--title` uses `ruamel.yaml` round-trip so user comments
  and formatting in `_quarto.yml` / `index.qmd` survive untouched. No
  "managed by matctl" marker comment needed — the `title:` key is the
  managed unit, scoped by filename.
