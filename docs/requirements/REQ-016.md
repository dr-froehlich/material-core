### REQ-016: Document identity fingerprint — per-project commit hash and template version in rendered output

- **Status:** DONE
- **Added:** 2026-05-20
- **Completed:** 2026-05-20
- **Verified by:** manual end-to-end smoke test 2026-05-20 (`matctl project add` → inspected `_quarto.yml` pre-render hook, `index.qmd` include, `.gitignore`; `matctl fingerprint` against a fresh git repo showed clean hash → `-dirty` suffix → `unknown` fallback paths; opt-out via `--no-fingerprint` toggled correctly through `project modify`)
- **Depends on:** REQ-013, REQ-014
- **Description:**

  When a rendered document (HTML book, single-file HTML, or PDF) is shared
  with a reviewer, there is currently no way for the reviewer to tell
  whether a later copy is the same artefact or has been revised. The
  source is versioned in git, but the rendered output carries no
  identifier that ties it back to a specific source state. This forces
  manual coordination ("the version I sent you on Tuesday") that breaks
  down as soon as the document is forwarded.

  Two facts together uniquely identify a rendered artefact:

  1. **The content commit** — which revision of the project's source in
     the `material` content repo was rendered. This changes when the
     author edits chapters, the index, or project-local assets.
  2. **The template version** — which `material-core` release supplied
     the brand SCSS, Typst show rules, and Quarto template fragments at
     render time. This changes when brand assets or template fragments
     are updated upstream and the pinned tag in
     `material/.github/workflows/publish.yml` is bumped.

  Either alone is insufficient: a brand-only change leaves the content
  commit unchanged, and a typo fix leaves the template version
  unchanged. Both together are a complete fingerprint — the same pair
  reproduces the same rendered output bit-for-bit (modulo non-
  deterministic PDF metadata).

  This requirement adds both identifiers to every rendered project as a
  standard, opt-out-able colophon field, surfaced in a location
  reviewers will notice (front matter or page footer, TBD in the
  planning step).

  Scope: rendering happens in CI. Local previews of uncommitted state
  are explicitly out of scope — the fingerprint may legitimately show
  the previous commit's hash for a working copy with unstaged edits,
  because the published artefact is always built from a committed
  state.

  Non-goals:

  - A semantic version counter (1.2.x with auto-incrementing patch).
    This was considered and rejected during scoping: the commit hash
    already answers "is this the same document?" and any version
    counter layered on top would either be noisy (bumped on every
    commit) or require manual curation (which the author has said they
    do not want as a default). If a curated version label is needed
    later, it can be added as an optional manifest field.
  - A content hash over the project's source tree. Considered, but
    rejected in favour of the per-project last-touching commit because
    (a) all published renders go through CI from a committed state, so
    git history is authoritative, (b) a commit hash is also a clickable
    reference back to the source, and (c) it avoids defining and
    maintaining a canonicalisation rule for what to hash.

- **Acceptance criteria:**
  - [x] Every project rendered through CI carries two identifiers visible to a reader of the output: the project's last-touching commit (short hash) and the `material-core` template version (the tag pinned in `publish.yml`).
  - [x] The commit hash reflects the **project's** last-touching commit (`git log -1 --format=%h -- <project>/`), not the repo HEAD. Edits to a sibling project must not change this project's hash.
  - [x] The template version reflects the actual installed `material-core` package version at render time (read via `importlib.metadata.version("material-core")` or equivalent), not a value duplicated in `projects.yml`.
  - [x] Both values are available to authoring sources as Quarto variables (e.g. `{{< var commit >}}` and `{{< var template >}}`) so they can be referenced anywhere in templates and `.qmd` content.
  - [x] The identifiers are rendered into the output by default for all three structures (`chapters`, `single`, `slides`) and all brands. Placement is consistent (front matter colophon for chapters/single; closing slide for slides — exact design in the planning step).
  - [x] Authors can suppress the fingerprint per project via a manifest flag (e.g. `fingerprint: false` in `projects.yml`) for use cases like camera-ready externally-archived documents.
  - [x] Local renders (`quarto preview`, `quarto render` outside CI) still produce a usable fingerprint: the commit hash is the project's last-touching commit if the working tree is clean, or that hash with a `-dirty` suffix if there are uncommitted changes inside the project directory. The template version is whatever `material-core` is installed (editable installs may report a dev version — acceptable).
  - [x] The pre-render hook fails gracefully if `git` is unavailable or the working directory is not a git checkout (e.g. a tarball build): both values fall back to `unknown` and rendering continues.
  - [x] No regression in existing CI render times: the hook adds well under one second per project.
  - [x] Documented in `material-core/CLAUDE.md` (mechanism, opt-out flag). Authoring-manual update in `material/material/chapters/01-authoring.qmd` is tracked separately in the `material` content repo (cross-repo update; not in this commit).

- **Notes:**

  Design context from the scoping conversation (2026-05-20):

  - The repo-wide HEAD commit was rejected as the content identifier
    because it changes when any project in `material` is edited, which
    is misleading for a reviewer of a single document.
  - A pure content hash over the project tree was rejected (see
    non-goals above). The user confirmed that publishing only happens
    via CI from committed state, so the "uncommitted preview" edge case
    that a content hash would handle better is not a real concern.
  - Brand and template assets live in `material-core`, a separate
    repository pinned by tag in `material/.github/workflows/publish.yml`.
    A commit-based identifier in `material` cannot detect upstream
    template changes, which is why the template version is surfaced as
    a second, independent field rather than folded into a single hash.
  - The `git log -1 -- <project>/` filter follows the path as recorded
    in `material`, not the symlinked brand assets that point into
    `material-core`. This is the desired behaviour: brand changes
    should show up as a template-version bump, not as a content-commit
    bump.

  Open questions for the planning step:

  - Exact placement of the fingerprint in each structure: title page
    colophon vs. footer vs. final page. May differ per structure.
  - Whether to also include the commit date alongside the hash for
    human readability (likely yes — `2026-05-20 abc1234` is more
    informative than `abc1234` alone).
  - Whether `material-core` should expose a small helper (e.g.
    `matctl fingerprint <project>`) that prints the resolved pair, both
    to support debugging and to give CI a single command to invoke from
    its pre-render step.
  - Whether the opt-out flag should be per-project only, or also
    per-group (so an entire group of externally-archived documents can
    suppress the fingerprint with one setting).
