### REQ-005: Standalone document support — `doc` template and `matctl doc add` / `doc remove`

- **Status:** OPEN
- **Added:** 2026-04-11
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-003, REQ-004
- **Description:**

Courses are multi-chapter books rendered with Quarto's book profile. Some
publications (the first concrete example is `esp-survival-guide`, currently
stranded as a chapter under `digital-und-mikrocomputertechnik`) are
standalone documents: single `.qmd` or small flat set of files, no slides, no
book-level TOC navigation. These need a distinct template and lifecycle
commands so they can be added, published, and removed without pretending to
be a course.

**Template — `material_core/templates/doc/`:**

Minimal Quarto project for a single document:

- `_quarto.yml` — declares the project and output formats (HTML and
  `orange-book-typst` PDF, no book profile, no slides), brand-linked via the
  same `_brand.yml` symlink mechanism as courses.
- `index.qmd` — document body with placeholders for title/author.
- `assets/` — empty directory for images/diagrams.
- No `chapters/`, no `slides/`, no `_shared/`.

Placeholders: `{{DOC_NAME}}`, `{{DOC_TITLE}}`, same substitution convention
as the course template (REQ-004).

**CLI — `matctl doc add <name> [--title ...]` and `matctl doc remove <name>`:**

Mirrors `course add` / `course remove` from REQ-004:

- Creates / deletes `./<name>/` from the doc template.
- Manifest entry uses `type: doc` (vs. `type: course`).
- Same idempotence, same confirmation, same `--yes` flag, same refusal
  conditions.

Implementation should share the scaffolder machinery with `course add`: one
internal helper that takes `(template_dir, project_type, name, title)` and
writes the files + patches the manifest. The `course` and `doc` click
commands are thin wrappers over it.

**CI support:**

REQ-003's matrix workflow already declares a `doc` branch (single `quarto
render <name>`, deploy `<name>/_output/*`, no slides). REQ-005 is the first
requirement that exercises it end-to-end. Part of the acceptance is running
a real `doc`-type project through the full CI matrix path and confirming
deploy lands at `material.professorfroehlich.de/<name>/`.

**Migrating `esp-survival-guide`:**

The existing `esp-survival-guide` content under
`digital-und-mikrocomputertechnik/` is flagged in `material/CLAUDE.md` as
"should be a standalone document". That migration is the natural end-to-end
test for this requirement, but is tracked separately as a content change
(not a tooling change) — this requirement only covers the mechanism, not the
specific migration. A dedicated content-side requirement or task can consume
REQ-005's output once it's ready.

- **Acceptance criteria:**
  - [ ] `material_core/templates/doc/` exists with `_quarto.yml`, `index.qmd`,
        and `assets/` — no book, chapters, or slides structure
  - [ ] `matctl doc add demo-doc` creates `./demo-doc/` from the template and
        appends `{name: demo-doc, type: doc}` to `projects.yml`
  - [ ] Scaffolded `demo-doc` builds successfully with `quarto render
        demo-doc` (HTML + PDF) with no manual editing
  - [ ] `matctl doc remove demo-doc --yes` cleanly removes directory and
        manifest entry
  - [ ] `course add` and `doc add` share an internal scaffolder helper (not
        two parallel implementations)
  - [ ] CI matrix workflow (REQ-003) successfully builds and deploys a
        `doc`-type project end-to-end, with output reachable under
        `material.professorfroehlich.de/<name>/`
  - [ ] `docs/administration.md` and `material/CLAUDE.md` document the
        course-vs-doc distinction and when to use each

- **Notes:**

- The doc template does not include a slides sub-project by design. If a
  specific document later wants slides, promote to a new requirement and
  decide whether to special-case it or introduce a third project type.
- Output directory layout for docs (`<name>/_output/*`) intentionally
  mirrors courses' book output (`<name>/_output/book/*`) minus the `book/`
  infix, so the Netcup deploy path stays flat per project.
- `esp-survival-guide` migration is the motivating real-world case but is
  explicitly out of scope for this requirement. Track the migration
  separately as a content task once the mechanism lands.
