### REQ-002: Session backlog: tooling ideas to refine post-split

- **Status:** DROPPED
- **Added:** 2026-04-11
- **Completed:** 2026-04-11 (all items resolved — promoted or dropped)
- **Verified by:** –
- **Depends on:** REQ-001
- **Description:**

Captures ideas discussed in the 2026-04-11 design session that should become
individual requirements after REQ-001 (the repo split) is complete. **This is
a placeholder, not implementation work.** Each item below should be promoted to
its own REQ-xxx (with full acceptance criteria) or explicitly rejected with
rationale before any implementation begins.

The reason this requirement exists at all is to prevent the design context
from being lost when this conversation's context is cleared. It will be marked
DROPPED (not DONE) once every item below is either promoted or rejected.

**Items to promote:**

1. **Python CLI `matctl`** — replaces `new-course.sh` and
   `scripts/manage-tokens.sh` with one tool. Likely subcommands:
   `course add`, `course remove`, `doc add`, `doc remove`,
   `token issue`, `token list`, `token revoke`, `publish`.
   Recommended stack: `typer` (CLI), `ruamel.yaml` (round-trips YAML
   comments — important for editing `_quarto.yml` files), `httpx`
   (Cloudflare API), `pyproject.toml` + `uv` for packaging.

2. **Matrix-driven CI** — replace per-course job blocks in `publish.yml`
   with one matrix job driven by an explicit manifest (e.g.
   `projects.yml`) or auto-discovery of `*/_quarto.yml`. Eliminates the
   manual workflow edit when adding a course/document — currently the
   most error-prone step in `docs/administration.md` §6.2.

3. **Standalone document support** — first-class `doc add` for
   non-lecture publications. Likely a different Quarto profile from
   courses (single document, not a book). Aligns with the broader
   "publishing platform, not just lectures" framing of REQ-001.

4. **Global admin token / cookie** — convenience for the operator
   (Peter) to bypass per-course token entry on his own devices. Two
   options:
   - (a) Use the existing `course="*"` wildcard token with a long
     expiry (zero code change — already supported per
     `docs/administration.md` §7.1).
   - (b) Add a separate `ADMIN_SECRET`-derived path in the Worker that
     issues an admin cookie surviving `COOKIE_SECRET` rotation.

   Decision deferred until the token CLI requirement is written.
   Recommendation leans toward (a) unless surviving secret rotation is
   important.

5. **Reusable GitHub Actions workflow** — alternative (or complement)
   to "install via pipx" in REQ-001. `material-core` exposes a callable
   workflow that `material` invokes via
   `uses: peter/material-core/.github/workflows/publish.yml@vX.Y.Z`.
   Cleaner than installing a CLI in CI; may not be enough on its own
   (local CLI use still needs the package).

6. **Shared assets sync mechanism** — how `_brand.yml`, `shared/base.scss`,
   fonts, and logo flow from `material-core` into a content build.
   Options: (a) `matctl init` / `matctl sync` copies them in, (b) git
   submodule, (c) CLI build wrapper symlinks them at build time, (d)
   installed Python package ships them and the CLI resolves their paths.
   Each has tradeoffs around editability during local Quarto preview.
   This is closely tied to REQ-001's open design question and may end up
   resolved there rather than as a separate REQ.

7. **Worker redeploy automation** — Worker source currently requires
   manual paste into the Cloudflare dashboard. Consider deploying via
   `wrangler` from `material-core` CI on tag push. Tradeoff: adds a
   Cloudflare API token to `material-core` repo secrets.

- **Resolution of each item (2026-04-11):**

  1. **Python CLI `matctl`** — promoted in pieces. Course lifecycle
     (`course add`/`remove`) → **REQ-004**. Document lifecycle
     (`doc add`/`remove`) → **REQ-005**. Token subcommands
     (`token issue/list/revoke`) **deferred** until after the content
     lifecycle work lands — will be promoted to a future REQ at that point.
     `matctl render` / `matctl deploy` / `publish` **dropped**: CI calls
     `quarto` directly and that layer is already as thin as it can be; a
     Python wrapper would create two code paths that can drift.
     `typer` recommendation **dropped**: staying on `click`, which the
     existing working commands already use.

  2. **Matrix-driven CI** — promoted to **REQ-003**. Uses an explicit
     `projects.yml` manifest (not auto-discovery) so that `matctl course
     add` / `doc add` have a single source of truth to patch.

  3. **Standalone document support** — promoted to **REQ-005**. Separate
     doc template and manifest type, sharing the scaffolder helper with
     `course add`.

  4. **Global admin token** — **deferred** alongside token CLI work.
     Once `matctl token issue` exists, the wildcard-course long-expiry
     token (option a) is a zero-code solution that can be documented in
     `administration.md` rather than needing its own code path.

  5. **Reusable GitHub Actions workflow** — **dropped**. The value of a
     reusable workflow is factoring out shared build logic; but once
     REQ-003 is in place, the per-repo `publish.yml` is already thin and
     the real build logic is in `quarto` + shell, not in code that could
     live in `material-core`. Indirection without reuse is a net loss.

  6. **Shared assets sync mechanism** — **resolved by REQ-001**. Option C
     (`matctl link` creates symlinks into the installed `material-core`
     package data) is implemented, tested, and documented. No follow-up
     requirement needed.

  7. **Worker redeploy automation** — **dropped**. The Worker is stable
     and changes maybe twice a year. Wiring `wrangler` into CI would
     require adding a Cloudflare API token to `material-core` repo
     secrets and building a deployment pipeline for something that takes
     30 seconds of dashboard paste when it actually changes. Revisit only
     if Worker edits become frequent.

- **Notes:**

- Reframe during promotion: REQ-002 was authored before REQ-001 landed and
  implicitly assumed a multi-repo content world. The actual split is single
  content repo (`material`) + single tooling repo (`material-core`), which
  simplifies several items. The content-lifecycle items (matrix CI, course
  add/remove, doc add/remove) form Phase 1 and were promoted. Token items
  form Phase 2 and are intentionally deferred until Phase 1 lands. Items
  without clear user value under the single-content-repo model were dropped
  with rationale above.
- Do not implement anything from this file directly — it is a DROPPED
  record of how the backlog was resolved, not a work item.
