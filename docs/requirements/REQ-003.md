### REQ-003: Matrix-driven CI with explicit `projects.yml` manifest

- **Status:** OPEN
- **Added:** 2026-04-11
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-001
- **Description:**

The current `material/.github/workflows/publish.yml` hard-codes one job block
per course. Adding a new course or document requires hand-editing three places
in the workflow file (the `changes` job outputs, the `paths-filter`
configuration, and a full copy of the render/deploy job block). This is the
most error-prone step in `docs/administration.md §6.2` and blocks the
content-lifecycle tooling in REQ-004 / REQ-005 from ever eliminating manual
workflow edits.

Replace the per-project job blocks with a single matrix-driven job that reads
an explicit manifest file, `projects.yml`, living at the root of the `material`
repository. The manifest enumerates every publishable project and its type;
the workflow builds only the projects whose files have changed in the push.

**Manifest format (`material/projects.yml`):**

```yaml
projects:
  - name: digital-und-mikrocomputertechnik
    type: course
  - name: esp-survival-guide
    type: doc
```

Fields:
- `name` — directory name under `material/` (also the URL path segment under
  `material.professorfroehlich.de/`).
- `type` — `course` or `doc`. Controls which render/deploy steps run:
  - `course`: `quarto render <name>` + `quarto render <name>/slides`; deploys
    `<name>/_output/book/*` and `<name>/slides/_output/*`.
  - `doc`: `quarto render <name>`; deploys `<name>/_output/*`. No slides step.

Explicit manifest is chosen over auto-discovery (`*/_quarto.yml` walk) because
the manifest is a maintenance tool: it is the single source of truth that
`matctl course add` / `matctl doc add` patches (REQ-004, REQ-005), it documents
intent (a half-finished directory is not accidentally published), and it
allows an ordered/filtered view if needed later.

**Workflow structure (target):**

1. **`changes` job** — uses `dorny/paths-filter` driven from `projects.yml` (or
   a small step that reads the manifest and emits a JSON list of project names
   whose paths changed in the push). Outputs a JSON array of project names to
   build.
2. **`build` job** — `strategy.matrix.project` expands over the JSON output
   from step 1. Guarded by `if: needs.changes.outputs.projects != '[]'`. Each
   matrix entry: checks out, installs `material-core`, `matctl link`, sets up
   Quarto + fonts, runs the type-specific render commands, deploys via
   `sshpass`+`scp` to Netcup.
3. **`workflow_dispatch`** — manual run builds everything in the manifest.

Only `publish.yml` and `projects.yml` live in `material`. `material-core` ships
no workflow file for this — the workflow stays thin and repo-local because
matrix semantics differ slightly between content repos and tooling repos, and
reusable-workflow indirection was rejected (see REQ-002 promotion notes).

- **Acceptance criteria:**
  - [ ] `material/projects.yml` exists with at least
        `digital-und-mikrocomputertechnik` registered as `type: course`
  - [ ] `material/.github/workflows/publish.yml` has exactly one `build` job
        using `strategy.matrix` over projects from the manifest (not one job
        per course)
  - [ ] Only projects whose files changed in the push are built; pushing a
        change to an unrelated file builds nothing
  - [ ] `workflow_dispatch` builds every project in the manifest
  - [ ] Adding a new course to `projects.yml` and creating its directory (no
        other workflow edits) causes it to be picked up and built on next push
  - [ ] Existing `digital-und-mikrocomputertechnik` build succeeds end-to-end
        (HTML + PDF + slides) with no visible regression vs. the pre-matrix
        workflow
  - [ ] Token issuance via `manage-tokens.sh` still works (unchanged by this
        requirement)
  - [ ] `docs/administration.md §6.2` updated to reflect the new "add a course"
        flow (edit `projects.yml`, don't edit the workflow)

- **Notes:**

- The `course` type render-step list is tied to the current course template
  structure (book + slides). If REQ-005's doc template arrives first, the
  workflow's `doc` branch can be added but untested until a real `doc`
  project exists.
- The `if: needs.changes.outputs.projects != '[]'` guard must be correct or
  the matrix job will fail on pushes that touch no project. Test this
  explicitly before marking the requirement DONE.
- Do not use a reusable workflow in `material-core` for this. That approach
  was considered and rejected in REQ-002 item 5 because the heavy build logic
  stays in `quarto` + shell, not in Python, so there is no shared code to
  factor out. The workflow in `material` is already thin.
