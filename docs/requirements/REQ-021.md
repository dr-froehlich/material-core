### REQ-021: De-quartizer — export a Quarto `.qmd` source tree to clean, self-contained GFM

- **Status:** OPEN
- **Added:** 2026-06-09
- **Completed:** –
- **Verified by:** –
- **Depends on:** –
- **Motivated by:** REQ-020 (external project manuals); useful well beyond it.
- **Description:**

  A general-purpose material-core tool that takes a Quarto `.qmd` source tree
  and emits a **plain GitHub-Flavored-Markdown mirror** that renders correctly
  in any standard markdown viewer (GitHub repo view, IDE preview, plain Pandoc)
  with **no Quarto runtime and no THD tooling**.

  The motivating consumer is REQ-020: a public project authors its handbook in
  first-class `.qmd` (for symbolic cross-references, callouts, includes), and
  the de-quartizer produces the GFM that a visitor to the *public* repo reads.
  But the tool is deliberately scoped as a standalone capability — any `.qmd`
  document can be "de-quartized" for distribution to readers who don't have
  Quarto. It is **not on the critical path for REQ-020's branded site** and may
  land after it; until it lands, public `.qmd` is readable-but-raw on GitHub.

  **Transforms (the Quarto specialties that don't degrade gracefully):**

  - **Cross-references** (`@sec-…`, `@fig-…`, `@tbl-…`, `[-@…]`) → resolved to
    working static links and/or numbers (e.g. `[Section 2.3](#engine)` or
    `[Figure 4](...)`). This is the crucial one: the GFM output keeps *working*
    references, just statically resolved instead of symbolic. Unresolvable refs
    must fail loudly, not emit raw `@…`.
  - **Callout blocks** (`::: {.callout-note|tip|warning|important|caution}`) →
    downgraded to blockquotes with a bold title line (e.g. `> **Note**`).
  - **Includes** (`{{< include … >}}`) → inlined (the included file's
    de-quartized content spliced in place).
  - **Conditional content** (`::: {.content-visible}` /
    `::: {.content-hidden}`, `unless-format`/`when-format`) → resolve for a
    plain-markdown target: keep "visible in all/this format", drop
    format-specific (e.g. revealjs-only) blocks.
  - **Other shortcodes** (`{{< video >}}`, `{{< embed >}}`, etc.) → degrade to a
    sensible plain-markdown equivalent or an explicit link; never leave raw
    `{{< … >}}` in the output.
  - **YAML front matter** → reduced to what a plain reader benefits from (title
    as an `# H1`, maybe author/date), Quarto-only keys stripped.
  - **Figure/table captions and labelled divs** → flattened to standard
    markdown images/tables with caption text preserved.

  **Pass-through (already valid GFM):** headings, lists, tables, fenced code,
  links, emphasis, blockquotes — emitted unchanged.

  **Scope guardrail:** the de-quartizer targets *correctness and readability*
  of the GFM, not visual fidelity to the branded HTML/PDF. Branding stays in
  the Quarto pipeline; the GFM mirror is intentionally generic.

  **Implementation strategy — SETTLED by spike (2026-06-09, Quarto 1.10.2 /
  Pandoc 3.8.3).** Build on `quarto render --to gfm` plus a thin
  post-processing filter. This is a thin filter layer, **not** a from-scratch
  AST tool. A throwaway spike rendering a representative `.qmd` (section + figure
  cross-refs, callout, include, conditional content, shortcode) to GFM showed
  Quarto natively handles most transforms:

  | Construct | Native `--to gfm` behaviour | Verdict |
  |---|---|---|
  | Cross-ref **numbering** (`@sec-`, `@fig-`) | Resolved to `Section 2`, `Figure 1` | ✅ reuse |
  | Callouts (`::: {.callout-note}`) | Lowered to GitHub `> [!NOTE]` admonition | ✅ native, minor title polish |
  | `{{< include >}}` | Inlined | ✅ |
  | Conditional content (`content-visible when/unless-format`) | Correctly kept/dropped for the gfm target | ✅ |
  | `{{< video >}}` and similar shortcodes | Degraded to a plain link | ✅ acceptable |

  **The three gaps the post-filter must close:**

  1. **Cross-refs are emitted as raw HTML, not GFM links** —
     `<a href="#sec-engine" class="quarto-xref">Section 2</a>` rather than
     `[Section 2](#engine)`. The filter rewrites these to markdown links.
  2. **Anchor mismatch breaks the links on GitHub** — the emitted
     `href="#sec-engine"` does *not* match GitHub's auto-generated heading slug
     (`# Engine` → `#engine`). For *heading* xrefs the filter must rewrite the
     href to GitHub's slug scheme. For *figure/table* xrefs there is no usable
     target at all (GitHub strips `id` from the `<div id="fig-…">` wrappers
     Quarto emits), so those degrade to the plain resolved number text
     (`Figure 1`) with no link.
  3. **Unresolved refs do not fail the build** — Quarto emits them as literal
     `?@sec-foo` and exits 0. The filter must scan for `?@` and enforce the
     "fail loudly" criterion itself.

  **Single-document authoring is the standard (decided 2026-06-09; see
  REQ-020).** `gfm` is **not a supported format for Quarto *book* projects** —
  `quarto render --to gfm` errors inside a book project, and even a single-file
  render inside a book-project directory inherits that and fails. The adopted
  answer is therefore upstream of this tool: project manuals are authored as a
  **single Quarto document** (optionally `{{< include >}}`-ing fragments), so
  the de-quartizer runs one `quarto render --to gfm` with **no chapter-assembly
  stage**. This is the common, supported path.

  **Fallback (book + assemble), documented but not the primary path:** if a
  specific manual is large enough to stay a book project, the de-quartizer must
  instead assemble its chapters into one combined document (with proper
  blank-line separators) and render that once — the spike confirmed
  cross-chapter references resolve this way, producing a single `HANDBOOK.md`.
  Per-file rendering in a non-book context is *not* viable: each file rendered
  in isolation loses cross-chapter ref resolution.

- **Acceptance criteria:**
  - [ ] A `matctl` command (name TBD, e.g. `matctl dequartize <path>
        [--out <dir>]`) takes a `.qmd` file or tree and writes a GFM mirror.
  - [ ] Heading cross-references resolve to working GFM links whose anchors
        match GitHub's slug scheme (verified clickable in GitHub's repo view);
        figure/table cross-references degrade to the resolved number text. No
        raw `quarto-xref` HTML `<a>` survives in the output.
  - [ ] The run scans for Quarto's unresolved-reference marker (`?@…`) and
        fails loudly with a clear error rather than shipping it to the reader.
  - [ ] Single-document input (the standard per REQ-020) renders to GFM with one
        `quarto render --to gfm` and no assembly stage. The book + assemble
        fallback, if implemented, preserves cross-chapter references; per-file
        isolated rendering is not used (book-project gfm is unsupported by
        Quarto).
  - [ ] Callouts, includes, conditional content, and known shortcodes are
        transformed per the Description; **no raw `:::` fences or `{{< … >}}`
        shortcodes appear in the output**.
  - [ ] Output is valid GFM that renders correctly in GitHub's repo view
        (verified on the `devsteward` handbook as the first real case).
  - [ ] Pass-through constructs (headings, lists, tables, code, links) are
        byte-faithful or semantically identical in the output.
  - [ ] The tool runs standalone — no THD brand assets or symlinks required to
        produce the GFM (branding is explicitly out of scope).
  - [ ] Round-trip on the `devsteward` handbook: the de-quartized GFM is
        committed into the public repo and renders as a usable manual with
        working internal links, satisfying REQ-020's public-reader goal.
  - [ ] `docs/authoring.md` documents the supported transforms, the unsupported
        constructs (if any), and how to invoke the de-quartizer.

- **Notes:**

  Source: decision 2026-06-09 to adopt B1 (canonical `.qmd`) for REQ-020 and
  split the de-quartizer into its own, separately-schedulable requirement
  because it is general-purpose tooling and a "nicety" for public repos rather
  than a blocker for the branded manuals site.

  Open decisions for a planning step:

  - **Where the GFM lands and who regenerates it.** Working model (from
    REQ-020): commit the generated GFM into the *public* repo so readers need
    no tooling; the `.qmd` source is the maintainer-facing canonical; the
    maintainer (who has material-core) runs the de-quartizer to regenerate.
    Decide whether regeneration is a manual `matctl` invocation, a pre-commit
    hook in the public repo, or a CI step in the public repo.
  - **AST vs source-rewrite strategy** — RESOLVED by the spike: build on
    `quarto render --to gfm` + thin post-filter (see Description). What remains
    for planning is the post-filter's host (Lua filter inside the render vs a
    standalone post-processing pass over the `.md`) and where the chapter
    assembly step lives.
  - **Cross-reference link targets** — RESOLVED: heading xrefs → GFM links with
    GitHub slugs; figure/table xrefs → resolved number text, no link.
  - **Failure policy** for shortcodes the tool doesn't yet understand: hard
    error (safe, default) vs pass-through-with-warning. (Unresolved *xrefs* are
    already decided: hard error on `?@`.)
