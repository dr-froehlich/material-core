### REQ-007: Group scope — shared access scope for co-deployed courses and documents

- **Status:** OPEN
- **Added:** 2026-04-13
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-005, REQ-006
- **Description:**

**Problem:** A course and one or more companion documents are logically related
and contain internal cross-links. Deploying them as separate top-level paths
(e.g. `/digital-und-mikrocomputertechnik/` and `/esp-survival-guide/`) means
each needs its own token. The Worker issues cookies scoped to the first URL
segment, so a cookie for one path does not cover the other. Cross-document
links fail for students who received only one token or who have not yet visited
both entry points.

**Solution:** Introduce the concept of a **group** — a shared URL-path scope
covering one or more co-deployed projects. Projects in the same group are
deployed under `<group>/<name>/` instead of `<name>/`. A single token issued
for the group covers every project beneath it. Tokens may also be issued for
a fully-qualified path (`<group>/<name>`) to grant access to one specific
document within a group.

The Cloudflare Worker is updated to perform **scope-based path-prefix
matching** instead of first-segment matching, making both token granularities
work correctly without breaking existing tokens.

**Scope of this requirement:** Worker authorization logic, CI deploy-path
changes, the `--group` flag on existing scaffold commands, and the `group:`
field in `projects.yml`. Terminology is fixed here (`group`, not `prefix`).

**Out of scope (covered by companion requirements):**
- The `type: group` entry in `projects.yml`, group lifecycle commands
  (`matctl group add/remove/modify`), title metadata, and `modify` support
  on courses/docs → **REQ-008**.
- Auto-generated group landing page at `<group>/index.html` → **REQ-009**.

---

### 1. `projects.yml` schema change

Add an optional `group` field to course/doc entries:

```yaml
projects:
  - name: digital-und-mikrocomputertechnik
    type: course
    group: mk4-26

  - name: esp-survival-guide
    type: doc
    group: mk4-26

  - name: unrelated-standalone-doc
    type: doc
    # no group — behaviour unchanged
```

The `group` value must satisfy the same regex as `name`: `[a-z0-9][a-z0-9._-]*`.

This requirement does not introduce a separate manifest entry for the group
itself; REQ-008 adds `type: group` entries with titles and existence-validation.
Implementations landing REQ-007 alone validate the `group` value only against
the slug regex.

---

### 2. `matctl course add` / `matctl doc add` — `--group` flag

Both scaffold commands gain an optional `--group` flag:

```bash
matctl course add digital-und-mikrocomputertechnik --group mk4-26
matctl doc add esp-survival-guide --group mk4-26
```

When supplied, `group: <value>` is written into the manifest entry alongside
`name` and `type`. Without `--group` the entry is identical to today.

---

### 3. Cloudflare Worker — scope-based authorization

**Current behaviour:** `course` = first path segment; `isAuthorized` checks
exact equality against `record.course`.

**New behaviour:** the token's `course` field is treated as a URL path scope
of arbitrary depth. Authorization passes when the request pathname starts with
`/<scope>/` (or equals `/<scope>`):

```js
function isAuthorized(record, pathname) {
  if (record.course === "*") return true;
  const prefix = "/" + record.course;
  return pathname === prefix
      || pathname.startsWith(prefix + "/");
}
```

The **session cookie** stores the token's scope verbatim (not the first
segment). `makeSessionCookie` receives the scope string from the token record;
`verifySessionCookie` checks that the request pathname is covered by the
cookie's stored scope using the same prefix test.

The KV field name remains `course` for backwards compatibility; no KV
migration. The field semantically represents a URL path scope.

**Backward compatibility:** existing tokens with `course:
"digital-und-mikrocomputertechnik"` satisfy `isAuthorized` for any path
starting with `/digital-und-mikrocomputertechnik/` — identical to today's
behaviour.

**Token granularities after this change:**

| Token `course` value | Covers |
|---|---|
| `mk4-26` | all of `/mk4-26/**` |
| `mk4-26/digital-und-mikrocomputertechnik` | only `/mk4-26/digital-und-mikrocomputertechnik/**` |
| `digital-und-mikrocomputertechnik` | `/digital-und-mikrocomputertechnik/**` (existing, unchanged) |
| `*` | everything (existing wildcard, unchanged) |

---

### 4. `matctl token issue` — iLearn URL derivation

No change to the KV write or the argument interface. The iLearn URL is derived
by splitting `course` on `/` and joining:

- `mk4-26` → `https://material.professorfroehlich.de/mk4-26/?token=...`
- `mk4-26/digital-und-mikrocomputertechnik` → `…/mk4-26/digital-und-mikrocomputertechnik/?token=...`
- `*` → `https://material.professorfroehlich.de/?token=...` (unchanged)

The existing special-case for `*` is retained; all other values use the generic
path-join rule.

---

### 5. CI `publish.yml` (in `material` repo) — deploy target

When a project has a `group`, the deploy target changes:

| `group` | Deploy target on Netcup |
|---|---|
| absent | `httpdocs/<name>/` (unchanged) |
| `mk4-26` | `httpdocs/mk4-26/<name>/` |

The workflow must:

1. Read the `group` field from `projects.yml` during matrix expansion.
2. Compute `DEPLOY_DIR = <group>/<name>` when group is set, else `<name>`.
3. Use `ssh mkdir -p httpdocs/<DEPLOY_DIR>/` before the `scp` step (the
   existing `rm -rf` + recreate pattern applies to `httpdocs/<DEPLOY_DIR>/`).
4. Deploy rendered output to `httpdocs/<DEPLOY_DIR>/` using the same rules as
   today (book output for `course`, flat output for `doc`).

Change detection is unaffected: project source files still live under `<name>/`
in the repository.

---

### 6. `matctl remove` — updated cleanup note

When a project has a group, the post-remove note updates the remote path from
`material.professorfroehlich.de/<name>/` to
`material.professorfroehlich.de/<group>/<name>/`.

- **Notes:**

- The `group` field is a deploy-time concept only. Quarto renders projects
  into `<name>/_output/...` as today; the group is invisible to Quarto.
- Cross-document links in content must be written as relative paths
  (`../esp-survival-guide/`) or absolute URLs. The group makes the cookie
  cover both paths; it does not rewrite links automatically.
- The auto-generated group landing page (`<group>/index.html`) fits naturally
  into this model: a static page deployed to `httpdocs/mk4-26/` is covered
  by the same `mk4-26` token with no further Worker changes. See REQ-009.
- The Worker update changes cookie internals (scope stored verbatim rather than
  derived from the first segment). Students with pre-existing cookies for
  root-level courses (e.g. `digital-und-mikrocomputertechnik`) will need to
  re-authenticate once after the Worker is redeployed. This is acceptable;
  document it in the deployment note for this requirement.
- No KV schema migration: the `course` field in existing token records is
  already a plain string. New values (`mk4-26`, `mk4-26/foo`) are just
  strings with a slash — no structural change. The KV field name is retained
  as `course` for backwards compatibility even though "scope" is more accurate.
- Terminology: **group** is the user-facing name for the grouping concept.
  Earlier drafts of this requirement used "prefix"; that term is reserved
  for the generic string-prefix matching performed by the Worker.
