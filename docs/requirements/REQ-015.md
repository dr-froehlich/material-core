### REQ-015: Make Mermaid → PDF (Typst) dependency explicit and self-installing

- **Status:** DONE
- **Added:** 2026-04-30
- **Completed:** 2026-04-30
- **Verified by:** `matctl doctor` smoke-tested on WSL2; chrome-headless-shell confirmed present
- **Depends on:** –
- **Description:**

  Mermaid blocks (` ```{mermaid} `) render correctly to PDF through the
  `orange-book-typst` format **today** — but only because Quarto's bundled
  Chrome client (`chrome-headless-shell`) is installed on the rendering
  machine. Quarto pre-renders each Mermaid block to a PNG via headless
  Chrome and embeds the PNG in the Typst output.

  The dependency is invisible:

  - CI installs it explicitly
    (`.github/workflows/publish.yml`: `quarto install chrome-headless-shell --no-prompt`).
  - Locally, it gets pulled in the first time someone runs
    `quarto install chrome-headless-shell` by hand.
  - **Nothing** in `material-core` (CLAUDE.md, matctl, authoring manual,
    administration manual) tells a new contributor that this step is
    required, nor enforces it.

  On a clean checkout without `chrome-headless-shell`, `quarto render`
  emits a warning and either skips or fails the diagram silently in a
  long render log — easy to miss until a student opens the PDF and finds
  the Mermaid figure missing. The same gap caused the confusion that
  triggered this requirement (a Claude session in `material/` reported
  Mermaid "doesn't render in Typst").

  This requirement closes the gap: document the prerequisite and make the
  local setup ergonomic so it can't be silently skipped.

- **Acceptance criteria:**
  - [ ] `matctl` exposes a way to ensure the Mermaid PDF dependency is satisfied locally — either a new `matctl doctor` command that checks and offers to install, or `matctl link` runs `quarto install chrome-headless-shell --no-prompt` (idempotent, fast no-op if already installed) on first use.
  - [ ] If the dependency is missing at render time and the project contains any `{mermaid}` block, the failure is loud, not silent — at minimum a documented `matctl` check that exits non-zero with a clear remediation message.
  - [ ] `material-core/CLAUDE.md` documents the chrome-headless-shell prerequisite for PDF rendering with Mermaid, and the command to install it locally.
  - [ ] `material/material-manuals/chapters/01-authoring.qmd §1.8 Diagrams (Mermaid)` notes that PDF rendering requires `quarto install chrome-headless-shell` on the local machine (CI handles it automatically).
  - [ ] No regression in the existing CI workflow — `publish.yml` continues to render Mermaid → PDF as today.
  - [ ] Verified by rendering a small project containing at least one `{mermaid}` block to `orange-book-typst` on a machine where `~/.local/share/quarto/chrome-headless-shell` has been removed: the failure mode is loud (matctl preflight, or a clearly identifiable render error), and re-running the documented setup step produces a PDF with the diagram embedded as an image.

  Design notes for the planning step:

  - Preferred direction: a new `matctl doctor` subcommand. Reasons:
    (a) keeps `matctl link` fast and side-effect-light, (b) gives a
    natural home for future preflight checks (font cache, Quarto version,
    pinned `MATERIAL_CORE_REF` mismatch, etc.).
  - Alternative: piggyback on `matctl link` with an opt-out flag. Faster
    to ship, but mixes concerns (linking is a per-project op, chrome is
    a per-machine op).
  - No alternate render path is needed. Considered pre-rendering Mermaid
    to SVG via a separate `mmdc` install or a Typst-native Mermaid
    package — both are strictly worse than what Quarto already does
    (extra tools, divergent CI, no parity with HTML output).
  - Open question: should `matctl doctor` also verify that the fonts
    listed in `shared/assets/fonts/` are installed in the local font
    cache? Out of scope here; track separately if it surfaces.
