### REQ-014: Multi-brand support — visual brand registry in `material-core`

- **Status:** OPEN
- **Added:** 2026-04-26
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-008
- **Description:**

  Today the `material` repository assumes THD branding everywhere: the
  brand file (`_brand.yml`), the SCSS in `shared/base.scss`, the logo,
  the favicon. The schutzkonzept project (a parish document, not THD
  lecture material) demonstrates a real need for non-THD branding.

  Introduce a brand registry inside `material-core` and let projects opt
  in to a brand at scaffold time. Brand scope is **strictly visual**:
  logo, primary colour, favicon, footer. No author identity, no per-brand
  language, no per-brand templates, no per-brand build flags. Anything
  beyond the visual surface stays brand-neutral and shared.

- **Acceptance criteria:**
  - [ ] `material_core/brands/<brand>/` exists for every shipped brand and contains exactly the visual assets in scope: logo, favicon, `_brand.yml` (which carries the primary colour and footer), and any brand-specific SCSS overlay needed to apply those choices.
  - [ ] Brand-neutral assets (theorems, mermaid defaults, fonts, base SCSS structure) live outside `brands/` — for example in `material_core/shared/` as today, with a documented rule that nothing brand-specific may live there.
  - [ ] At least two brands ship: `thd` (current behaviour, default for backwards compatibility) and a neutral brand (`none` or `generic`) suitable for non-THD documents like the schutzkonzept project.
  - [ ] Brand selection is recorded per project in `projects.yml` (e.g. `brand: thd`).
  - [ ] `matctl link` resolves brand-specific symlinks based on the per-project `brand:` field; running `matctl link` in a project directory wires up the brand the project was created with.
  - [ ] `matctl course modify` / `doc modify` (or their REQ-013 successor) accept `--brand <name>` to switch a project's brand; symlinks and template references are updated; landing pages regenerated.
  - [ ] Existing projects without a `brand:` field continue to render identically to today (implicit `thd`).
  - [ ] Group landing pages (REQ-009) render correctly when their children mix brands; resolution rule is documented (proposed default: landing page is brand-neutral, regardless of children).
  - [ ] `docs/administration.md` documents the brand model, the visual-only scope, the folder layout, and how to add a new brand.
- **Notes:**

  Source: `material/material-core-input.md` §2 + §5; decisions 2026-04-26:

  - Scope: visual only (logo, primary colour, favicon, footer). No
    quick-fix THD-logo path patch — this requirement is the only place
    brand assets are addressed.
  - Storage: brands live in `material-core` (versioned with tooling).
    No support for ad-hoc brand paths in `material/` in v1.
  - Folder structure carries the limited visual scope: a brand directory
    contains *only* what visually changes, nothing else.

  Open design questions for the planning step:

  - Group brand semantics: does a group itself carry a `brand:` field
    (so the landing page can match), or is the landing page always
    neutral? Current proposal: always neutral; cheaper and avoids
    ambiguity when children mix brands.
  - Worker / domain coupling (input §5): out of scope for this
    requirement. Track as a follow-up if a need surfaces.
  - Splitting: consider REQ-014a (registry + ship `thd` and one neutral
    brand + scaffold-time `--brand`) and REQ-014b (`modify --brand`
    switching) if the single requirement gets too large during planning.
