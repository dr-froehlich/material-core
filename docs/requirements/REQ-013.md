### REQ-013: Orthogonal universal document creation — chapters × slides × brand

- **Status:** OPEN
- **Added:** 2026-04-26
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-005, REQ-012, REQ-014
- **Description:**

  Today the system has two scaffold commands frozen as combinations:
  `course add` (multi-chapter + slides, THD-branded) and `doc add`
  (single-file, no slides, THD-branded). The schutzkonzept project
  exposed combinations the system can't express cleanly: a multi-chapter
  document without slides and without THD branding.

  Evolve the creation interface into three orthogonal CLI options on a
  single universal creation flow:

  1. **Structure:** single-file vs multi-chapter book.
  2. **Slides:** with or without an accompanying slides project.
  3. **Brand:** which brand to apply (see REQ-014).

  Plus the existing `--lang` (REQ-010), `--title`, `--subtitle`, `--group`.

  Any combination of the three orthogonal flags must be scaffoldable. The
  `type` field in `projects.yml` and the directory layout reflect the
  chosen combination; a downstream consumer can always reconstruct the
  three choices from the manifest.

- **Acceptance criteria:**
  - [ ] A single creation command (e.g. `matctl project add <name>`, exact name TBD) accepts orthogonal flags: `--chapters/--single`, `--slides/--no-slides`, `--brand <name>`, plus existing `--lang`, `--title`, `--subtitle`, `--group`.
  - [ ] All eight combinations of the three orthogonal flags scaffold successfully and `quarto render` produces the expected output for each.
  - [ ] `projects.yml` records the choices in a way that (a) lets `matctl` regenerate landing pages without re-inspecting on-disk files and (b) is human-readable.
  - [ ] Existing `matctl course add` and `matctl doc add` either become thin shortcuts to specific combinations (`course = --chapters --slides --brand thd`, `doc = --single --no-slides --brand thd`) or are deprecated with a migration note in the changelog. Decision documented in the requirement notes before implementation.
  - [ ] `matctl course modify` / `doc modify` (or their successor) can change brand and structure-relevant fields where physically meaningful; impossible transitions (e.g. flipping `--chapters/--single` after content exists) fail with a clear message rather than corrupting the project.
  - [ ] Slides scaffolding lives in a separate `slides/` subtree (matching today's course template) and is omitted entirely when `--no-slides` is selected — no empty `slides/` directory left behind.
  - [ ] Templates live under `material_core/templates/` in a structure that reflects the orthogonal axes (e.g. composable fragments rather than one template per combination); duplication across the eight combinations is minimised.
  - [ ] `docs/authoring.md` and `CLAUDE.md` document the orthogonal model with a short decision matrix.
- **Notes:**

  Source: `material/material-core-input.md` §6; decision 2026-04-26
  (option (c): three orthogonal CLI choices on a universal creation
  process).

  This is the largest behavioural change in the post-schutzkonzept
  batch. Recommend a planning step (`docs/plans/req-013-…md`) before
  implementation to settle:

  - Command name and migration story for `course add` / `doc add`.
  - Template composition strategy: one template per combination
    (8 templates, much duplication) vs a fragment/overlay system
    (1 base + structure overlay + slides overlay + brand overlay).
    Strong preference for the latter to keep maintenance tractable.
  - How `projects.yml` records the three axes (booleans? a single
    `type` enum of eight values? separate keys?).
  - Whether group landing pages need to know about the axes (probably
    only `brand`, for visual consistency).

  Depends on REQ-014 because `--brand` is one of the three orthogonal
  choices; REQ-014 must define what valid brand values are before this
  requirement can finalise its CLI surface. Depends on REQ-012 because
  any multi-chapter scaffold variant inherits the unnumbered-preface
  fix.
