### REQ-001: Split repository into `material-core` (tooling) and `material` (content)

- **Status:** DONE
- **Added:** 2026-04-11
- **Completed:** 2026-04-11
- **Verified by:** End-to-end build + deploy through the new split (CI run 24280114365 on `dr-froehlich/material` green, 2m03s); live-edit validation of `material_core/shared/base.scss` visible in rendered HTML with no re-install/re-link; token issue/validate/revoke round-trip against the unchanged Cloudflare Worker + KV
- **Depends on:** –
- **Description:**

The current `vorlesungen` repository mixes tooling (Cloudflare Worker, deploy
scripts, shared brand assets, CI workflow, course scaffolder) with content
(lecture chapters, slides). Projected scale is 6 lectures plus ~10 standalone
publications per year, all with frequent edits. In a single repo, code-related
commits become invisible in the noise of content edits, making the tooling
history unreviewable a year later. Branches are not a robust separation
mechanism (easy to commit to the wrong one by accident). The clean fix is to
split by lifecycle: code in one repo, content in another.

**Target structure:**

- **`material-core`** — new repo. Owns all engineering work:
  - Python CLI `matctl` (to be built in subsequent requirements)
  - Cloudflare Worker source (`cloudflare/worker.js`)
  - Reusable GitHub Actions workflow / workflow templates
  - `_brand.yml`, `shared/` (SCSS, fonts, logo, diagram assets)
  - Course/document scaffolder (replaces `new-course.sh`)
  - All engineering documentation: `docs/administration.md`, `docs/authoring.md`,
    `docs/plans/`, `docs/requirements/` (including this requirements system)
  - Tagged with semver releases (`v1.0.0`, etc.). No PyPI publish — content
    repos install directly from a pinned git tag via `pipx`/`pip`.

- **`material`** — current `vorlesungen` repo, renamed on GitHub. Owns only
  content:
  - Lecture courses and standalone publication projects as sibling top-level
    directories
  - Tiny `.github/workflows/publish.yml` that pins a `material-core` version,
    installs it, and delegates the build/deploy
  - GitHub Actions secrets: `SSH_HOST`, `SSH_USER`, `SSH_PASSWORD`
  - No tooling files, no Worker source, no shared assets in the repo itself
    (those flow in from `material-core` at build time)

**Shared infrastructure (unchanged by the split):**

- One Cloudflare Worker
- One KV namespace (`LECTURE_TOKENS`)
- One subdomain (`material.professorfroehlich.de`)
- One token namespace — student URLs work regardless of which repo a
  course/document originated in
- The Worker routes by URL path; it does not know or care about repo
  provenance

**Naming decisions:**

- `material` for content (matches the subdomain, doesn't lock to lectures)
- `material-core` for tooling (clarity of relationship beats generic names
  like `pubctl` or `material-tools`)
- CLI binary `matctl` (short, typeable)

**Why now, before any other automation work:**

Migrating content out of a 5-year-old monorepo is much harder than starting
separated. All subsequent requirements (Python CLI, matrix CI, etc.) will be
authored and implemented against `material-core`, so the split is a hard
prerequisite.

- **Notes:**

- The mechanism for sharing `_brand.yml` and `shared/` across the repo boundary
  is the main open design question and should be the focus of the
  implementation plan. Local-preview ergonomics (editing `shared/base.scss`
  while running `quarto preview`) is a key constraint — whichever mechanism is
  chosen must not require a `material-core` re-release for every SCSS tweak.
- This requirement intentionally does not specify the CLI design — that is
  REQ-002's territory and beyond.
- Solo user, no collaborators to coordinate with. The split exists for
  *future-self* readability of code history, not for team workflow.
- Git history housekeeping (`git filter-repo`) is explicitly rejected as a
  long-term strategy: it rewrites history, breaks existing clones, and breaks
  GitHub issue/PR cross-references. Prevention via the split is the right
  approach.
- After REQ-001 is DONE, this requirements directory itself lives in
  `material-core`. The next requirement to be authored (promoting items from
  REQ-002) should be created there, not here.
