### REQ-018: Multilingual projects — a project may carry the same content in more than one language

- **Status:** OPEN
- **Added:** 2026-05-26
- **Completed:** –
- **Verified by:** –
- **Depends on:** REQ-008, REQ-009, REQ-013
- **Description:**

  Today every project entry in `projects.yml` has a required scalar
  `lang: de|en`. A project is, by construction, monolingual. Some
  documents (e.g. `richtlinie-abschlussarbeiten`) need to ship in
  multiple languages, and some projects that start monolingual will
  later evolve to bilingual.

  Language is already a first-class field on every project. This REQ
  lifts it from a scalar to a plural: a project may speak one language
  (today's behavior, unchanged) or several (new). No new orthogonal
  concept is introduced — no "variant", no "slug". A multilingual
  project is just a project that has more than one language.

  **Manifest shape.** `lang` and `languages` are mutually exclusive on
  a `type: project` entry:

  ```yaml
  # monolingual (today's shape, unchanged):
  - name: mk4-ss26
    type: project
    structure: chapters
    slides: true
    brand: thd
    lang: de
    title: Mikrocomputertechnik 4
    subtitle: SS 2026

  # multilingual:
  - name: richtlinie-abschlussarbeiten
    type: project
    structure: single
    slides: false
    brand: thd
    languages:
      de:
        title: Richtlinie für Abschlussarbeiten
      en:
        title: Thesis Guidelines
  ```

  Per-language keys are `title` (required) and `subtitle` (chapters
  only, optional). All other project fields — `structure`, `slides`,
  `brand`, `group`, `fingerprint` — are project-level and shared
  across languages.

  The two shapes are normalized: a `languages:` map must contain at
  least two entries, and a single-language project must use the
  `lang:` scalar form. This rules out a "multilingual project with
  one language" state that would otherwise duplicate the monolingual
  case and complicate every code path that branches on shape.

  **On-disk layout.** Mirrors the deploy URL.

  - Monolingual: `<name>/` is the Quarto project root, exactly as today.
    `<name>/index.qmd` (single), `<name>/chapters/...` (chapters),
    `<name>/slides/...` (when slides), one set of brand symlinks at
    `<name>/`.
  - Multilingual: `<name>/<lang>/` is each language's Quarto project
    root. Each `<lang>/` subdir is a standalone Quarto project with
    its own `_quarto.yml`, its own `index.qmd` / `chapters/` /
    `slides/`, and its own set of five brand symlinks (or six with
    slides — see REQ-017). The parent `<name>/` holds an
    auto-generated `index.html` language-chooser landing page; it is
    not itself a Quarto project.

  **Deploy URL.** Mirrors the on-disk layout. Monolingual:
  `<group?>/<name>/`. Multilingual: `<group?>/<name>/<lang>/`.

  **Tokens.** No CLI default change needed. The Cloudflare Worker's
  `isAuthorized` is a path-prefix match (`pathname` startsWith
  `/<scope>/`), so a token issued for `<name>` already authorizes
  every language subdir under it. Adding a language to an existing
  monolingual project does not break existing tokens. The token
  redemption cookie is scoped the same way, so sessions survive too.

  Per-language scoping is opt-in: `matctl token issue <name> --lang
  <lang>` mints a token scoped to `<name>/<lang>` only, useful for
  handing a German cohort a German-only key.

  **Mono → multi migration is the headline verb.** A single
  `matctl project lang add <name> <new-lang> --title "..."` call:

  1. Moves the existing flat project tree into `<name>/<existing-lang>/`
     (`index.qmd`, `chapters/`, `slides/`, `_quarto.yml`, all brand
     symlinks, `.gitignore`, `_variables.yml` if fingerprint is on).
     Uses `git mv` for tracked files so history follows; untracked
     source files in the project tree are moved alongside;
     Quarto-generated artifacts (`_book/`, `_freeze/`, `_output/`,
     `*_files/`) are not moved — they regenerate at the new location
     on next render and are gitignored at each layer.
  2. Rewires the brand symlinks under `<name>/<existing-lang>/` to point
     to the brand registry the same way as before.
  3. Scaffolds `<name>/<new-lang>/` from the same template fragments,
     with brand symlinks wired.
  4. Generates `<name>/index.html` (the language-chooser landing).
  5. Rewrites the manifest entry from `lang: <code>` to the
     `languages: {...}` shape, with titles preserved for the
     existing language and the new one taken from the CLI flag.
  6. If the project sits under a group, regenerates the group's
     landing page (the project's link there still points to
     `<name>/`, which now serves the chooser).

  **Multi → mono** (`matctl project lang remove <name> <lang>`): if
  removing the language would leave one survivor, the operation
  collapses the survivor's subdir contents back up to `<name>/` and
  rewrites the manifest entry back to `lang: <surviving>` shape.
  Removing a non-last language just deletes that `<lang>/` subdir,
  drops the row from `languages:`, and regenerates the chooser.

  **Scope:**

  - Manifest schema: `lang` scalar XOR `languages` map on
    `type: project` entries; load-side validation rejects entries
    that set both or neither.
  - `matctl project add`: unchanged in shape and semantics. Result is
    monolingual; `--lang <code>` stays required.
  - New `matctl project lang add <name> <lang> --title "..."
    [--subtitle "..."]`: scaffold a new language. If the project is
    currently monolingual, also perform the mono→multi migration above.
  - New `matctl project lang remove <name> <lang> [--yes]`: remove a
    language. If only one survives, collapse back to monolingual flat
    layout.
  - New `matctl project lang list <name>`: convenience listing.
  - `matctl project modify --title "..."` on a multilingual project
    requires `--lang <code>` to disambiguate which language's title
    to update; without it, refuse.
  - `matctl project modify --brand <newbrand>`: re-seats symlinks
    under every language subdir.
  - `matctl project modify --slides`: applies to every language subdir
    (false→true scaffolds `slides/` in each, true→false rejects if any
    has content).
  - `matctl project modify --fingerprint/--no-fingerprint`: applies
    to every language subdir.
  - `matctl project modify --group <newgroup>`: as today; the
    multilingual project moves as a unit. Regenerates the old and
    new group landings.
  - `matctl project remove <name> [--yes]`: removes the whole
    `<name>/` tree (chooser, all language subdirs, all their
    content) and drops the manifest entry. Symmetric with how
    today's flat remove deletes the single project directory.
  - `matctl fingerprint <project>` on a multilingual project
    requires `--lang <code>` to disambiguate; without it, the
    command prints one block per language. `--write` requires
    `--lang` because it targets one `_variables.yml`. The
    per-language `_quarto.yml`'s pre-render hook invokes
    `matctl fingerprint --write` from within `<name>/<lang>/`, so
    no `--lang` is needed at render time — cwd resolves it.
  - `matctl link` / `matctl unlink`: fan out the symlink set under
    every language subdir for multilingual projects. From inside
    `<name>/` runs the project-wide fanout; from inside
    `<name>/<lang>/` runs that language only.
  - `matctl token issue <name> [--lang <lang>]`: optional `--lang`
    for per-language scoping. Without it, scope is `<name>` (covers
    all languages).
  - Auto-generated `<name>/index.html` chooser, generated by the
    existing landing-page generator parameterized for "project with N
    languages" (reuses the brand-aware layout used for groups).
  - Fingerprint colophon (REQ-016): each language subdir is an
    independent Quarto project with its own pre-render hook, own
    `_variables.yml`, own inline colophon. Editing only the German
    chapters flips only the German colophon's commit hash.

  **Non-goals:**

  - Per-language brand override. A multilingual project has one
    brand; mixing thd/generic across languages is a non-goal here.
    The `languages:` shape leaves room to add `brand:` per language
    later if needed.
  - Per-language `slides:` or `structure:` variance. Slides toggle
    and structure are project-wide.
  - In-page language switcher in the rendered HTML (so a reader can
    hop de↔en from the current page). That is a Quarto template
    concern, not a manifest concern. Possible follow-up REQ.
  - Cross-language sibling linking inside chapter `.qmd` files
    (same-page-different-language hrefs). Quarto-template concern.
  - Renaming the URL segment per language (e.g. German uses
    `richtlinien/abschluss/de/` but English uses
    `guidelines/thesis/en/`). The URL segment is always the ISO
    language code under `<name>/`.

- **Notes:**

  **Why this design, after several wrong turns (2026-05-26).**

  The original framing was "how do we let two projects share short
  URL paths so a bilingual document can deploy at `<group>/de/` and
  `<group>/en/`." First attempt introduced `slug:` decoupled from
  `name:`, which conflated two different things — `name` is already
  the machine-name; "slug" was a misnomer — and required group
  semantics to bend (slugs unique only within a group, on-disk
  layout still flat, no cross-group reuse). Second attempt
  ("variants") added a `variant_of:` back-reference or a
  `variants:` list, which described what bilingual projects *are*
  but was still a graft: a wrapper concept for the real underlying
  property that each project has a language.

  The realization: `lang` was already first-class on every project.
  The clean move is not to add a wrapper concept but to let `lang`
  be plural. A project speaks one or more languages, and when it
  speaks more than one, languages get subdirectories. This is how
  every i18n routing system on the web models the same problem.

  Consequences that fell out of the cleaner model:

  - No new orthogonal concept. `lang` scalar XOR `languages` map is
    the only schema change.
  - On-disk = URL. Each language subdir is a normal Quarto project;
    no deploy-time path rewriting.
  - Tokens unchanged. Worker's path-prefix match already covers
    sub-language paths. Mono→multi does not break existing tokens.
  - Mono→bi is one CLI verb: `matctl project lang add`. Same verb
    handles every subsequent language too.
  - Groups orthogonal. A multilingual project may or may not sit
    under a group; the group does not need to know about languages.
  - The bilingual-shared-prefix use case that drove the original
    slug framing — `richtlinie-abschlussarbeiten/de/` and
    `…/en/` deploying side-by-side under one short prefix —
    no longer requires a group. One multilingual project at the
    repo root produces exactly that URL shape. Groups remain
    useful for bundling unrelated co-deployed material, which is
    what they were designed for.

  **Prior partial implementation.** A `slug:` field was added in
  commit e388990 ("feat(matctl): decouple project name from on-disk
  slug (REQ-018, partial)") on 2026-05-26 and reverted the same day
  via `git reset --hard HEAD~1` after the conceptual rethink. The
  slug commit shipped no tagged release (last tag at revert was
  v0.9.1) and was never pushed to origin. The slug concept is gone
  from the design.

  **Token migration safety.** Worker code at
  `material_core/cloudflare/worker.js:95-99`:

  ```js
  function isAuthorized(record, pathname) {
    if (record.course === "*") return true;
    const prefix = "/" + record.course;
    return pathname === prefix || pathname.startsWith(prefix + "/");
  }
  ```

  Pure prefix check. A token with scope `richtlinie-abschlussarbeiten`
  authorizes `/richtlinie-abschlussarbeiten/`,
  `/richtlinie-abschlussarbeiten/de/...`,
  `/richtlinie-abschlussarbeiten/en/...`, and any other
  sub-language. So a project gaining its second language does not
  require any token to be revoked or reissued. Same prefix logic
  applies to the session cookie's scope check
  (`worker.js:142-147`), so issued sessions survive the migration
  too.

  **Open template question (out of scope for this REQ, may surface
  during implementation).** Quarto crossref labels are
  language-keyed by `lang:` in `_quarto.yml`. Since each language
  subdir is an independent Quarto project, each carries its own
  `lang:` value and gets the right crossref labels automatically.
  No template change required for that. The "language switcher in
  rendered HTML" feature is separate and deferred.

  in a follow-up session before any code is written, and will cover
  manifest-loader changes, the mono→multi migration internals, CLI
