### REQ-020: External project manuals — surface a public project's handbook under the `manuals` group from a single canonical source

- **Status:** DONE (material-core tooling; live-site deploy pending the `material` PR — Phase 9)
- **Added:** 2026-06-09
- **Completed:** 2026-06-13 (v0.10.0)
- **Verified by:** Local end-to-end acceptance against `dr-froehlich/devsteward`
  (branded HTML + PDF render, committed-wrapper/gitignored-content split,
  fetch idempotency, re-pin, manifest validation) — see plan Phase 7.
- **Depends on:** REQ-005, REQ-009, REQ-013
- **Related:** REQ-021 (de-quartizer — completes the public-reader story; may land after this)
- **Description:**

  Some of Peter's software projects are public (first concrete example:
  [`devsteward`](https://github.com/dr-froehlich/devsteward), whose handbook
  lives at `devsteward/handbook/*.md` inside the project repo). Two
  requirements pull on that handbook at once:

  1. **Surface it on the manuals site.** The handbook should appear under the
     existing `manuals` group on `material.professorfroehlich.de`, rendered
     through the standard branded `matctl`/Quarto/CI pipeline, alongside
     `material-manuals`, `examengineer-manuals`, `thermo-manual`, `memzy`.

  2. **Keep the public project self-contained.** The public repo must ship a
     usable manual on its own. It must not depend on `material` (a private
     repo) being present in order to have documentation. A visitor to the
     public repo must be able to read the manual with no THD tooling.

  These two goals agree on one thing: the **canonical source must live in the
  public project repo**, not in `material`. `material` is therefore a
  *consumer*, never the owner, of this content. The requirement defines (A) how
  `material` pulls the content in and wraps it for the branded site, and (B)
  the source format that lets the same content serve both a branded Quarto site
  and a plain public reader.

  This is a new, reusable project category — call it an **external manual** (or
  "shared manual"): a manuals-group project whose content is owned by a
  different repository. It generalises beyond `devsteward` to any public
  project that wants its handbook on the manuals site.

  **(A) Transport — how `material` consumes external content**

  **Decided (2026-06-09): A2 — build-time fetch via `matctl`.** A1 is recorded
  below as the rejected alternative for context.

  The hard constraint is CI, not local preview. Locally, a symlink into a
  sibling working tree (the mechanism `matctl link` already uses for `shared/`
  and `_brand.yml`) would work. But `material`'s GitHub Actions runner has no
  sibling checkout on disk, so a working-tree symlink renders locally and
  **breaks in CI/deploy**. The bridge must be git-based. Two candidate
  mechanisms:

  - **A1 — git submodule (rejected).** Add the external repo (or just its handbook
    subtree) as a submodule under a manuals-group project; `material` commits
    only a thin Quarto wrapper plus a pinned commit pointer. No content
    duplicated in `material`'s tree; explicit version pin (a feature for a
    *published* manual — see the freshness decision in Notes). Cost:
    submodule-aware checkout in CI (`submodules: true`) and the usual submodule
    papercuts. Pragmatic, zero new `matctl` tooling.

  - **A2 — build-time fetch via `matctl` (CHOSEN).**
    Extend the pattern the repo already lives by. A `projects.yml` entry of a
    new type declares an external source, e.g.:

    ```yaml
    - name: devsteward-manual
      type: external-manual      # exact key TBD
      group: manuals
      brand: pf
      source: git@github.com:dr-froehlich/devsteward.git
      path: handbook
      ref: v0.3.0                # pinned tag — see freshness decision (pin, don't track main)
      structure: single          # external manuals are authored as ONE Quarto document
      lang: en
    ```

    **Authoring contract — external manuals are single Quarto documents, not
    book projects (decided 2026-06-09).** The public project authors its
    handbook as a single `.qmd` (optionally assembled from per-topic fragments
    via `{{< include >}}`, which keeps the source files separate while still
    rendering as one document). Rationale: Quarto does not support `gfm` output
    for *book* projects, so a single document is what lets the de-quartizer
    (REQ-021) emit clean GFM with one `quarto render --to gfm` and no chapter
    assembly stage. Cross-references resolve fully within a single document, so
    nothing is lost on the branded side; the only trade is the book's
    multi-page chapter navigation, which is an acceptable (often preferable)
    loss for a reference handbook. A genuinely oversized manual may fall back to
    book + assemble (see REQ-021), but single-doc is the standard.

    `matctl link` (locally) and a CI pre-render step clone/fetch the source at
    `ref`, drop the handbook into a **gitignored, generated** project dir, and
    wrap it with the branded Quarto shell. `material` commits only the manifest
    reference and the wrapper — **zero handbook content**, no pin drift in the
    tree. This is the natural extension of the existing "external content
    pulled in and gitignored" philosophy (`shared/`, `_brand.yml`). Cost: real
    `matctl` feature work; CI needs read access to the source repo (same owner;
    SSH key already present).

  In both cases the **branded Quarto wrapper lives in `material`, not in the
  public repo** — Quarto config, brand selection, `index.qmd`/book front
  matter, group membership. That keeps the public repo free of THD brand
  tooling and preserves goal 2.

  **(B) Source format — first-class Quarto vs plain markdown**

  **Decided (2026-06-09): B1 — canonical `.qmd`.** The de-quartize export that
  completes the public-reader story is split out into **REQ-021** and may land
  *after* this requirement (it is a nicety for the public repo, not on the
  critical path for the branded manuals site). Sequencing implication: see the
  interim-usability note below and in the acceptance criteria.

  `.qmd` is a strict superset of GFM, but the
  high-value Quarto features are exactly the ones that **do not degrade
  gracefully** in a plain renderer:

  - Cross-references (`@sec-engine`) render as raw text in any non-Quarto
    viewer. GitHub renders neither these nor `.qmd` files at all (it shows
    `.qmd` as source).
  - `{{< include >}}`, `::: {.callout-*}`, `::: {.content-visible}` show raw
    fences/shortcodes.

  Therefore **"symbolic cross-references in the committed source" and "the
  committed source renders correctly for a public reader on GitHub" are
  mutually exclusive in the same file.** The two coherent architectures:

  - **B1 — Canonical `.qmd` + de-quartize export (CHOSEN).** Author in
    first-class `.qmd`: full cross-references, callouts, includes — satisfying
    the stated core benefit (resolved chapter/section references instead of
    hard-coded numbers). A standard material-core **de-quartizer** (tracked
    separately as **REQ-021**) produces a GFM mirror that *resolves*
    cross-references into static links/numbers, downgrades callouts to
    blockquotes, and inlines includes. The public reader consumes the GFM
    mirror (committed into the public repo so a GitHub visitor needs nothing);
    the branded site consumes the `.qmd` directly. Net: symbolic authoring
    **and** plain-markdown usability, with references surviving as statically
    resolved links.

  - **B2 — Canonical plain `.md`, "Quarto-safe" conventions, no de-quartizer
    (rejected).** Author in pure GFM that renders natively on GitHub today;
    Quarto reads the same `.md` through the branded pipeline. Simplest, most
    self-contained, zero conversion tooling. Casualty: symbolic
    cross-references. Rejected because the brief is explicitly forward-looking
    and names cross-references a core benefit.

  **Interim usability (REQ-020 lands before REQ-021).** Because the
  de-quartizer is deferred, between REQ-020 shipping and REQ-021 landing the
  public repo carries `.qmd` source that GitHub displays as **raw source, not
  rendered**. This is the accepted, temporary state: the branded manuals site
  (goal 1) is fully served from day one; the *public reader's* experience
  (goal 2) is degraded to readable-but-raw until REQ-021 supplies the GFM
  mirror. Acceptable because the public manual is a nicety relative to the
  branded deliverable.

- **Acceptance criteria:**
  - [x] A public project's handbook appears under the `manuals` group on
        `material.professorfroehlich.de`, branded and rendered via the standard
        CI pipeline, with `devsteward` as the first real case. *(Tooling +
        local branded render verified; live-site deploy lands with the
        `material` PR — Phase 9.)*
  - [x] The canonical source of the handbook lives in the public project repo;
        no handbook content is the *authoritative* copy inside `material`.
        *(`git status` shows only `_quarto.yml`, `.gitignore`, the Typst
        render scaffold, and the manifest entry — zero handbook bytes.)*
  - [x] The public repo, checked out alone with no THD tooling, presents a
        readable manual to a visitor. Interim (pre-REQ-021): raw `.qmd` source,
        readable but unrendered on GitHub. Full (post-REQ-021): de-quartized GFM
        mirror that renders natively. The interim state is explicitly accepted.
  - [x] `material` obtains the content through a git-based mechanism that works
        in GitHub Actions CI, not only in local preview (no reliance on a
        sibling working-tree symlink). *(`matctl external fetch --all` shallow-
        clones the pinned ref; HTTPS source needs no CI secret.)*
  - [x] The branded Quarto wrapper (format, brand, group membership) lives in
        `material`/`material-core`, leaving the public repo free of brand
        tooling.
  - [x] `projects.yml` records the external-manual project such that landing

  - [ ] The de-quartize export (REQ-021) produces GFM in which Quarto

  - [~] `docs/administration.md` and `docs/authoring.md` document the

  **Decisions made 2026-06-09:**

  - **Source format → B1** (canonical first-class `.qmd`). The de-quartizer is
    **split into REQ-021** as general-purpose tooling and may land *after* this
    requirement; until it does, the public repo's reader experience is the
    accepted interim raw-`.qmd` state (see Description §B).

  - **Transport → A2** (`matctl` build-time fetch). A1 (submodule) rejected.
    Open implementation detail: confirm `material`'s CI has read access to the
    external repo (same owner; SSH key present) and decide where the fetched,
    gitignored project dir is generated.

  - **Freshness → pin a tag.** Do not track `main`. The external manual
    publishes from a pinned tag (`ref:` in the manifest), bumped deliberately,
    so a mid-refactor doc never auto-publishes to the manuals site.

  - **Where the de-quartized GFM lives → public repo** (decided under REQ-021):
    commit the generated GFM into the *public* repo so readers need no tooling,
    with the `.qmd` source as the maintainer-facing canonical. Captured here for
    context; specified in REQ-021.

  **Implementation note (2026-06-13): devsteward ref to pin.** The single-doc
  conformant layout (`handbook.qmd` + five `_NN-*.qmd` fragments) is committed
  on devsteward's `dev` branch (`fd70ebc`), but the only existing **tag**,
  `v0.1.0`, predates it and still carries the old five-`.md`-file layout. The
  freshness decision (pin a tag) therefore requires **cutting a new devsteward
  tag** at/after `fd70ebc` before the `material` PR (Phase 9) can pin a tag
  rather than the `dev` branch. The material-core tooling is ref-agnostic and
  was verified against `fd70ebc` (sha — exercises the bare-sha clone fallback)
  and `dev`.

  **Remaining open detail for a planning step (`docs/plans/req-020-…md`):**

  - **Relationship to REQ-013.** The external-manual category is a new
    project shape; decide whether it is a `type`/axis within REQ-013's
    orthogonal model or a sibling project type. Likely it reuses REQ-013's
    structure/brand axes and adds an `external source` dimension on top.

  - **Brand for `devsteward-manual`.** Other personal-project manuals use
    `brand: pf` (`material-manuals`, `memzy`); default this one to `pf` unless
    a project-specific brand is wanted.
